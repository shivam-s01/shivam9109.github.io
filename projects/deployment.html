Overview

Why Two Servers?

Free-tier cloud platforms like Railway and Render spin down idle servers after inactivity — cold starts can take 15–30 seconds. A single-server architecture means users hit that cold start directly. The dual-server approach solves this.

Both Railway (primary) and Render (fallback) run identical Flask apps, deployed from the same GitHub repo via separate connected projects. The client-side JavaScript races both servers simultaneously on first load — whichever responds first becomes the active server for that session.

A self-ping keep-alive scheduler runs on each server, calling its own /health endpoint every 10 minutes to prevent cold starts. SSRF guards on the proxy endpoint prevent the ping mechanism from being weaponised to probe internal networks.

Deployment Flow

From Code to Live in Seconds

📝 Code Push to GitHub Trigger
Developer pushes to main branch from Termux on Android. GitHub receives the push event and fires webhooks to both cloud providers simultaneously.
⚙️ Parallel Build Railway Render
Both platforms pull the latest commit, install Python dependencies from requirements.txt, and run gunicorn app:app as the entry point. Build logs are available in each platform's dashboard.
🔐 Environment Variables Injected Secure
Both platforms have identical env vars set in their dashboards: RAPIDAPI_KEY, PROXY_SECRET, ONESIGNAL_APP_ID, ONESIGNAL_KEY. These never touch the codebase or git history.
🏁 Health Check + Switchover Live
Client JS calls /health on both servers with a 6-second race timeout. First to respond wins. Railway typically wins due to faster cold starts. Render is the silent backup.
🔄 Keep-Alive Loop Continuous
Each server pings its own /health every 10 minutes via a background APScheduler job. This prevents the 30s cold-start penalty. SSRF allowlist ensures only localhost pings are permitted.

Code Sample

Keep-Alive Scheduler

# app.py — self-ping keep-alive with SSRF guard from apscheduler.schedulers.background import BackgroundScheduler import requests, os ALLOWED_PING_HOST = "127.0.0.1" # SSRF: only localhost allowed def keep_alive(): port = os.environ.get("PORT", "5000") url = f"http://{ALLOWED_PING_HOST}:{port}/health" try: r = requests.get(url, timeout=8) if r.status_code == 200: print("[keep-alive] OK") except Exception as e: print(f"[keep-alive] Failed: {e}") scheduler = BackgroundScheduler() scheduler.add_job(keep_alive, "interval", minutes=10) scheduler.start() # Health endpoint @app.route("/health") def health(): return {"status": "ok", "server": os.environ.get("SERVER_ID", "unknown")}, 200
Features

Full Capability Set

🏁
Client-Side Server Race
Frontend fires parallel fetch to both server URLs. A Promise.race picks the winner. The losing request is aborted via AbortController.
💓
Self-Ping Keep-Alive
APScheduler background job pings own /health every 10 minutes. Keeps the dyno warm. SSRF allowlist is enforced at the URL level.
🔄
Automatic Failover
If the primary server is down or slow, the fallback takes over silently. No user-visible error, no manual intervention needed.
🚀
Push-to-Deploy CI/CD
GitHub webhooks trigger builds on both platforms simultaneously on every push to main. Zero manual deployment steps.
🔑
Secrets Management
All API keys and secrets live in platform dashboard env vars. Git history is clean. Rotating a key takes 30 seconds per platform.
🛡️
SSRF Hardening
Proxy endpoints validate destination hostnames against an allowlist. Internal IPs (10.x, 172.x, 192.168.x, 127.x) are blocked unconditionally.
